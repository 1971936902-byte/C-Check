# C-Check 新服务器部署配置说明文档

本文档说明在新服务器 `git clone` C-Check 仓库后，如何复制配置文件、修改关键变量，并执行一键部署。

## 1. 使用场景

适用于原生 Linux 部署方式：

- FastAPI 后端运行在 Python 虚拟环境中。
- Celery worker 使用 systemd 管理。
- MySQL、Redis、Nginx 使用服务器本机服务。
- 前端 Vue 项目构建为静态文件，由 Nginx 提供访问。
- VLLM 模型服务可以单独部署，也可以在同一台服务器上注册到 C-Check。

推荐系统：

- Ubuntu 22.04 / Ubuntu 24.04
- root 或 sudo 权限
- 已开放 Web 映射端口
- 如需本机部署模型，服务器需要 NVIDIA GPU、驱动和足够显存

## 2. 首次部署命令

推荐使用真正的一键部署入口 `provision`。新服务器只需要 clone 仓库，然后执行一条命令：

```bash
git clone https://github.com/1971936902-byte/C-Check.git /opt/c-check
cd /opt/c-check

sudo DEPLOY_ENV=/etc/c-check/c-check.env \
  PUBLIC_HOST_ALT=180.127.11.177 \
  bash deploy/native/c-check-deploy.sh provision 223.109.239.36 13958 8800
```

这条命令会自动完成：

1. 生成 `/etc/c-check/c-check.env`。
2. 自动生成数据库密码、JWT 密钥和管理员初始密码。
3. 写入 `WEB_PORT=8800`、`PUBLIC_ORIGIN=http://223.109.239.36:13958` 和 CORS。
4. 安装系统依赖、Python/Node 环境、后端依赖。
5. 初始化 MySQL、Redis、Alembic 数据库迁移。
6. 构建前端。
7. 写入 systemd 和 Nginx 配置。
8. 启动 API、worker、Nginx、MySQL、Redis，并尽量启动已有 VLLM 服务。
9. 执行健康检查。

参数含义：

- `223.109.239.36`：主要公网 IP 或域名。
- `13958`：云平台提供的公网 Web 端口。
- `8800`：云平台映射到服务器内部的 Web 端口。
- `PUBLIC_HOST_ALT=180.127.11.177`：可选的备用公网 IP，会自动加入 CORS。

如果服务器已经部署过，只是重启、SSH 端口变化或云平台端口映射变化，优先执行：

```bash
cd /opt/c-check
sudo DEPLOY_ENV=/etc/c-check/c-check.env bash deploy/native/c-check-deploy.sh recover
```

`recover` 会自动重写应用 `.env`、systemd、Nginx，启动服务并做健康检查，不需要手工改 Nginx。

下面的手动复制配置方式仅作为需要完全自定义时的备用方案。

```bash
git clone https://github.com/1971936902-byte/C-Check.git /opt/c-check
cd /opt/c-check

mkdir -p /etc/c-check
cp deploy/native/c-check.env.example /etc/c-check/c-check.env
vim /etc/c-check/c-check.env

sudo DEPLOY_ENV=/etc/c-check/c-check.env bash deploy/native/c-check-deploy.sh install
```

其中真正需要你修改的是：

```bash
/etc/c-check/c-check.env
```

## 3. 必须修改的配置项

### 3.1 仓库与安装目录

```dotenv
REPO_URL=https://github.com/1971936902-byte/C-Check.git
BRANCH=master
APP_DIR=/opt/c-check
```

说明：

- `REPO_URL`：项目 GitHub 地址，一般不用改。
- `BRANCH`：部署分支，生产环境通常使用 `master`。
- `APP_DIR`：服务器上的项目目录，推荐 `/opt/c-check`。

### 3.2 Web 访问地址

```dotenv
WEB_PORT=8800
PUBLIC_ORIGIN=http://YOUR_SERVER_IP:8800
CORS_ORIGINS=["http://YOUR_SERVER_IP:8800","http://localhost"]
```

说明：

- `WEB_PORT`：Nginx 在服务器内部监听的端口。
- `PUBLIC_ORIGIN`：浏览器实际访问地址。
- `CORS_ORIGINS`：允许访问后端 API 的前端来源，必须包含 `PUBLIC_ORIGIN`。

示例：

如果云平台公网地址是：

```text
http://180.127.11.167:24164/
```

并且云平台将公网 `24164` 映射到服务器内部 `8800`，则配置为：

```dotenv
WEB_PORT=8800
PUBLIC_ORIGIN=http://180.127.11.167:24164
CORS_ORIGINS=["http://180.127.11.167:24164","http://localhost"]
```

注意：

- `WEB_PORT` 写服务器内部端口。
- `PUBLIC_ORIGIN` 写浏览器访问的公网地址。
- 如果有多个公网 IP 或域名，都要加入 `CORS_ORIGINS`。

### 3.3 后端 API 监听地址

```dotenv
BACKEND_HOST=127.0.0.1
BACKEND_PORT=8000
```

说明：

- 一般不需要修改。
- 后端只监听本机 `127.0.0.1`，由 Nginx 代理 `/api/`。
- 不建议直接把 FastAPI 端口暴露到公网。

### 3.4 MySQL 数据库

```dotenv
MYSQL_DATABASE=c_check
MYSQL_USER=c_check
MYSQL_PASSWORD=CHANGE_ME_USE_A_STRONG_DATABASE_PASSWORD
MYSQL_ROOT_LOGIN="mysql -uroot"
```

说明：

- `MYSQL_DATABASE`：数据库名。
- `MYSQL_USER`：应用使用的数据库账号。
- `MYSQL_PASSWORD`：应用数据库密码，必须改成强密码。
- `MYSQL_ROOT_LOGIN`：部署脚本用来创建数据库和用户的 root 登录命令。

常见情况：

如果 MySQL root 无密码：

```dotenv
MYSQL_ROOT_LOGIN="mysql -uroot"
```

如果 MySQL root 有密码：

```dotenv
MYSQL_ROOT_LOGIN="mysql -uroot -p你的root密码"
```

如果使用 sudo socket 登录：

```dotenv
MYSQL_ROOT_LOGIN="sudo mysql"
```

### 3.5 Redis

```dotenv
REDIS_URL=redis://127.0.0.1:6379/0
```

说明：

- 默认即可。
- Celery worker 使用 Redis 做任务队列。

### 3.6 JWT 与管理员账号

```dotenv
JWT_SECRET=CHANGE_ME_USE_A_LONG_RANDOM_SECRET_AT_LEAST_32_CHARS
JWT_EXPIRE_MINUTES=480
ADMIN_USERNAME=admin
ADMIN_PASSWORD=CHANGE_ME_USE_A_STRONG_ADMIN_PASSWORD
```

说明：

- `JWT_SECRET`：登录令牌签名密钥，至少 32 个字符，必须随机。
- `JWT_EXPIRE_MINUTES`：登录有效期，默认 480 分钟。
- `ADMIN_USERNAME`：首次初始化管理员用户名。
- `ADMIN_PASSWORD`：首次初始化管理员密码，至少 12 个字符。

示例生成随机密钥：

```bash
openssl rand -hex 32
```

注意：

- 不要使用示例里的 `CHANGE_ME`。
- 管理员首次登录后，建议立刻在个人中心修改密码。

### 3.7 上传限制

```dotenv
UPLOAD_MAX_FILE_BYTES=1048576
UPLOAD_MAX_ARCHIVE_BYTES=10485760
UPLOAD_MAX_EXTRACTED_BYTES=10485760
UPLOAD_MAX_FILES=200
UPLOAD_MAX_ARCHIVE_ENTRIES=1000
UPLOAD_MAX_PATH_LENGTH=512
STORAGE_PATH=/opt/c-check/uploads
```

说明：

- `UPLOAD_MAX_FILE_BYTES`：单个 `.c/.h` 文件最大大小。
- `UPLOAD_MAX_ARCHIVE_BYTES`：上传 zip 压缩包最大大小。
- `UPLOAD_MAX_EXTRACTED_BYTES`：zip 解压后总大小限制。
- `UPLOAD_MAX_FILES`：一次审查最多 C 源文件数量。
- `UPLOAD_MAX_ARCHIVE_ENTRIES`：zip 内最多条目数。
- `UPLOAD_MAX_PATH_LENGTH`：文件路径最大长度。
- `STORAGE_PATH`：上传文件保存目录。

默认值适合轻量团队使用，一般不用改。

### 3.8 Mock 与模型重试

```dotenv
MOCK_MODEL_ENABLED=false
MODEL_MAX_ATTEMPTS=3
MODEL_MAX_TOKENS=2048
MODEL_STRUCTURED_OUTPUTS_ENABLED=true
ALLOW_INSECURE_DEFAULTS=false
```

说明：

- `MOCK_MODEL_ENABLED=false`：生产环境必须关闭 mock 模型。
- `MODEL_MAX_ATTEMPTS=3`：模型返回格式错误时最多重试 3 次。
- `MODEL_MAX_TOKENS=2048`：单次模型审查允许的最大输出 token 数。该值需要给输入提示词和 C 代码预留上下文空间；4K 上下文模型建议保持 2048，长上下文模型可按需调高。
- `MODEL_STRUCTURED_OUTPUTS_ENABLED=true`：启用 JSON Schema 结构化输出约束。推荐保持开启；只有在模型服务不支持 `response_format=json_schema` 时才临时改为 `false`。
- `ALLOW_INSECURE_DEFAULTS=false`：禁止弱密码、弱密钥启动。

生产环境不要设置：

```dotenv
ALLOW_INSECURE_DEFAULTS=true
```

### 3.9 Python 与 Node.js

```dotenv
PYTHON_BIN=python3
PYTHON_VERSION=3.12
NODE_MAJOR=22
NODE_VERSION=22.21.1
NODE_DIST_URL=https://npmmirror.com/mirrors/node
NODE_INSTALL_DIR=/opt/nodejs/22.21.1
```

说明：

- `PYTHON_BIN`：服务器 Python 命令。
- `PYTHON_VERSION`：后端虚拟环境使用的 Python 版本，Ubuntu 22.04 默认 Python 3.10 时脚本会用 uv 自动安装 3.12。
- `NODE_MAJOR`：Node.js 主版本，推荐 22。
- `NODE_VERSION`：Node.js tarball 兜底安装版本，推荐使用已验证的 22 LTS 版本。
- `NODE_DIST_URL`：Node.js tarball 下载源。国内服务器推荐 `https://npmmirror.com/mirrors/node`，海外服务器也可改成 `https://nodejs.org/dist`。
- `NODE_INSTALL_DIR`：Node.js tarball 解压安装目录。

部署脚本会优先通过 NodeSource 安装 Node.js 22。如果云服务器访问 NodeSource 超时，会自动改用 `NODE_DIST_URL` 下载 tarball 并创建 `node`、`npm`、`npx` 命令链接。后端要求 Python 3.12 及以上，脚本会在系统 Python 不满足时自动安装 uv 并创建 Python 3.12 虚拟环境。

## 4. VLLM 模型节点配置

如果模型服务已经单独部署，且提供 OpenAI-compatible API，可以让部署脚本自动注册模型节点。

```dotenv
REGISTER_VLLM_MODEL=true
VLLM_DISPLAY_NAME=Qwen2.5 Coder 32B AWQ
VLLM_MODEL_IDENTIFIER=qwen2.5-coder-32b-awq
VLLM_BASE_URL=http://127.0.0.1:8001
VLLM_API_KEY=你的VLLM_API_KEY
VLLM_TIMEOUT_SECONDS=600
```

说明：

- `REGISTER_VLLM_MODEL=true`：部署或更新时自动写入模型节点。
- `VLLM_DISPLAY_NAME`：前端显示名称。
- `VLLM_MODEL_IDENTIFIER`：VLLM 的 served model name，必须和 `/v1/models` 返回的 id 一致。
- `VLLM_BASE_URL`：模型服务地址，不要带 `/v1`。
- `VLLM_API_KEY`：VLLM 鉴权 key。
- `VLLM_TIMEOUT_SECONDS`：模型调用超时时间，30B+ 模型建议 600 秒。

如果暂时不注册真实模型：

```dotenv
REGISTER_VLLM_MODEL=false
```

部署完成后，也可以在管理后台手动新增模型节点。

### 4.1 管理后台模型目录与一键部署

本项目新增了“模型部署”管理页，管理员可以从模型目录中选择模型并创建下载/部署任务。当前内置目录位于：

```text
deploy/models/catalog.json
```

默认支持：

- DeepSeek-Coder-14B-Instruct
- CodeLlama-13B-Instruct
- StarCoder2-15B

如果后续要新增模型，优先修改 `deploy/models/catalog.json`，补充 `key`、`display_name`、`model_identifier`、`huggingface_repo`、`modelscope_repo`、`default_port`、`default_served_model_name` 等字段即可；后端无需改 Python 代码，管理后台会自动读取新目录。

后端自动执行下载/部署脚本默认关闭，避免在本地 Windows 或无 GPU 服务器误触发大模型下载。Linux GPU 服务器确认 Docker、NVIDIA Container Toolkit、VLLM 镜像、磁盘和显存准备好后，可以在 `/etc/c-check/c-check.env` 中增加：

```dotenv
MODEL_CATALOG_PATH=/opt/c-check/deploy/models/catalog.json
MODEL_DEPLOYMENT_ENABLED=true
MODEL_DEPLOYMENT_SCRIPT=/opt/c-check/deploy/models/deploy-vllm-model.sh
VLLM_IMAGE=vllm/vllm-openai:latest
VLLM_API_KEY=请替换为模型服务key
HF_TOKEN=请替换为HuggingFace token
TENSOR_PARALLEL_SIZE=1
GPU_MEMORY_UTILIZATION=0.90
MAX_MODEL_LEN=8192
```

如果 HuggingFace 下载不稳定，管理后台创建部署任务时选择 `modelscope` 来源；脚本会设置 `VLLM_USE_MODELSCOPE=true` 并使用目录中的 `modelscope_repo`。如果 `MODEL_DEPLOYMENT_ENABLED=false`，后台不会真正执行下载，会在部署记录日志中返回一条可手动执行的脚本命令。

注意事项：

- 13B/14B/15B 模型仍然很大，部署前确认磁盘空间、显存和带宽。
- 多个大模型不一定能在单 GPU 上同时运行，建议每个模型使用不同端口，并按显存情况逐个启动。
- 自动登记的模型节点使用 `served_model_name` 作为审查请求中的模型标识，必须与 VLLM `/v1/models` 返回的 id 一致。

## 6. 部署后检查命令

查看服务状态：

```bash
sudo DEPLOY_ENV=/etc/c-check/c-check.env bash /opt/c-check/deploy/native/c-check-deploy.sh status
```

查看 API 日志：

```bash
sudo DEPLOY_ENV=/etc/c-check/c-check.env bash /opt/c-check/deploy/native/c-check-deploy.sh logs c-check-api
```

查看 worker 日志：

```bash
sudo DEPLOY_ENV=/etc/c-check/c-check.env bash /opt/c-check/deploy/native/c-check-deploy.sh logs c-check-worker
```

手动检查网页：

```bash
curl -I http://127.0.0.1:8800/
```

手动检查 API：

```bash
curl http://127.0.0.1:8800/api/models
```

未登录时返回 `Not authenticated` 是正常的，说明 API 已经被 Nginx 代理到后端。

## 7. 更新代码

以后代码更新后，在服务器执行：

```bash
cd /opt/c-check
sudo DEPLOY_ENV=/etc/c-check/c-check.env bash deploy/native/c-check-deploy.sh update
```

该命令会：

1. 拉取 GitHub 最新代码。
2. 重写应用 `.env`。
3. 安装/更新 Python 后端依赖。
4. 执行 Alembic 数据库迁移。
5. 重新构建前端。
6. 重启 API、worker 和 Nginx。

## 8. 备份

```bash
sudo DEPLOY_ENV=/etc/c-check/c-check.env bash /opt/c-check/deploy/native/c-check-deploy.sh backup
```

备份内容包括：

- MySQL 数据库 SQL 文件
- 上传目录压缩包
- 部署配置文件
- 应用 `.env`

备份目录：

```text
/opt/c-check/backups/<时间戳>/
```

## 9. 常见问题

### 9.1 提示 CHANGE_ME 未替换

说明配置文件里还有占位符。执行：

```bash
grep CHANGE_ME /etc/c-check/c-check.env
```

把所有结果替换成真实强密码或随机密钥。

### 9.2 浏览器打不开网页

检查：

```bash
systemctl status nginx --no-pager
ss -ltnp | grep 8800
curl -I http://127.0.0.1:8800/
```

如果服务器内部正常，但公网打不开，通常是云平台端口映射或安全组没有开放。

### 9.3 登录失败

检查：

- `ADMIN_USERNAME`
- `ADMIN_PASSWORD`
- API 服务是否 active
- 数据库迁移是否完成

```bash
systemctl status c-check-api --no-pager
cd /opt/c-check/backend
/opt/c-check/.venv/bin/alembic current
```

### 9.4 模型健康检查失败

检查 VLLM：

```bash
systemctl status c-check-vllm --no-pager
curl -H "Authorization: Bearer <VLLM_API_KEY>" http://127.0.0.1:8001/v1/models
```

确认：

- `VLLM_BASE_URL` 不要带 `/v1`
- `VLLM_MODEL_IDENTIFIER` 等于 `/v1/models` 返回的模型 id
- `VLLM_API_KEY` 和模型服务启动参数一致

### 9.5 模型返回格式错误

C-Check 已支持：

- 后端严格 JSON schema 校验
- 失败后自动重试
- 前端查看模型日志

可以在前端任务状态或历史报告中点击“模型日志”查看模型原始返回内容。

## 11. 服务器重启或 SSH 连接变化后的快速恢复

当云服务器重启、SSH 端口变化、预留端口变化后，不要直接假设旧的访问方式仍可用。建议按下面顺序恢复。

### 11.1 确认 SSH 与公网映射

先用云平台给出的最新 SSH 命令连接，例如：

```bash
ssh root@223.109.239.36 -p 13912
```

同时确认 Web 端口映射，例如：

```text
公网 13958 -> 服务器内网 8800
```

如果云平台写的是“外网端口 -> 内网端口”，浏览器访问要带外网端口，例如：

```text
http://223.109.239.36:13958/
http://180.127.11.177:13958/
```

### 11.2 检查服务是否已经恢复

```bash
cd /opt/c-check
git log -1 --oneline
systemctl is-active c-check-api c-check-worker nginx mysql redis-server c-check-vllm
ss -ltnp | grep -E ':(8800|8000|8001) '
```

期望：

- `c-check-api` 为 `active`
- `c-check-worker` 为 `active`
- `nginx` 为 `active`
- `mysql` 为 `active`
- `redis-server` 为 `active`
- `c-check-vllm` 为 `active`
- Nginx 监听 `0.0.0.0:8800`
- API 监听 `127.0.0.1:8000`
- VLLM 监听 `127.0.0.1:8001`

### 11.3 确认 `/etc/c-check/c-check.env`

云平台映射到内网 `8800` 时，配置应类似：

```dotenv
WEB_PORT=8800
PUBLIC_ORIGIN=http://223.109.239.36:13958
CORS_ORIGINS='["http://223.109.239.36:13958", "http://180.127.11.177:13958", "http://223.109.239.36", "http://180.127.11.177", "http://localhost", "http://127.0.0.1:18000"]'
```

检查命令：

```bash
grep -E '^(WEB_PORT|PUBLIC_ORIGIN|CORS_ORIGINS)=' /etc/c-check/c-check.env
```

注意：不要把真实密码、Token、API Key 写入 Git 文档。

### 11.4 如果 Nginx 没有监听 8800

```bash
grep -q 'listen 0.0.0.0:8800;' /etc/nginx/sites-available/c-check.conf \
  || sed -i '/listen 0.0.0.0:80;/a\    listen 0.0.0.0:8800;' /etc/nginx/sites-available/c-check.conf

nginx -t
systemctl reload nginx
curl -I http://127.0.0.1:8800/
```

### 11.5 重启核心服务

```bash
systemctl restart c-check-api c-check-worker
systemctl reload nginx
systemctl is-active c-check-api c-check-worker nginx mysql redis-server c-check-vllm
```

VLLM 大模型服务启动会比普通 Web 服务慢。`systemctl is-active c-check-vllm` 变成 `active` 后，还可以继续确认接口：

```bash
curl -i http://127.0.0.1:8001/v1/models
```

如果返回 `401 Unauthorized`，说明 VLLM 已经响应，只是需要 API Key。

### 11.6 Windows 本地浏览器隧道

如果本地使用 `http://127.0.0.1:18000/` 访问云服务器，服务器重启或 SSH 端口变化后，旧隧道可能失效。

检查端口占用：

```powershell
Get-NetTCPConnection -LocalPort 18000 -ErrorAction SilentlyContinue
```

如果旧进程占用但访问超时，结束旧进程：

```powershell
Stop-Process -Id <旧进程ID> -Force
```

重新建立隧道：

```powershell
Start-Process -FilePath ssh.exe -ArgumentList @(
  '-N',
  '-L','18000:127.0.0.1:8800',
  '-o','ExitOnForwardFailure=yes',
  '-o','StrictHostKeyChecking=no',
  '-p','13912',
  'root@223.109.239.36'
) -WindowStyle Hidden
```

检查：

```powershell
curl.exe -I http://127.0.0.1:18000/admin
```

### 11.7 本次踩坑总结

- SSH 端口从旧端口切换到 `13912`，旧连接方式不可再用。
- 云平台公网端口是 `13958`，服务器内部 Web 端口是 `8800`，两者不能混淆。
- `/etc/c-check/c-check.env` 里如果仍是 `WEB_PORT=80`，一键部署脚本会把 Nginx 写回 80，导致公网映射到 `8800` 时访问失败。
- Windows PowerShell 会提前解析某些 Bash 写法，例如 `$(date ...)`、`|| true` 和复杂 JSON 引号。复杂配置建议用远程 `python3` 修改，写完后用 `grep` 回读。
- `c-check-vllm` 启动最慢，进入 active 后还要等模型 API 初始化。`/v1/models` 返回 `401 Unauthorized` 是正常响应，不是服务不可用。
